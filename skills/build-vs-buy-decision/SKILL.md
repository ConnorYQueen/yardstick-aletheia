# Build versus buy decision

What it does: decides, for one workflow, whether to build the autonomous system, buy a vendor's, or compose - so you own what encodes your judgment and rent only what you safely can.

When to run it: before committing to any autonomous system, and again at renewal.

Steps:
1. Run the five tests on the workflow:
   - Differentiation: does this capability set you apart, or is it commodity.
   - Data ownership: does it run on your proprietary data, or the vendor's.
   - Switching cost and exit: if you leave the vendor later, do you keep your data and your substrate.
   - Total cost of ownership: model build and buy over a multi-year horizon, not year one. Build it as a sheet from the report's ROI model.
   - Hollow-out test: if this vendor disappeared, are you crippled or merely inconvenienced.
2. Read the answers together. Buy when it is commodity, the vendor's data advantage is real, time to value dominates, and exit is clean. Build when it is core, runs on your data, or vendor dependence would hollow you out.
3. When the answer is mixed - it usually is - compose: buy the substrate and the models, build the orchestration, the agent spec, the evals, and the governance.
4. Whatever you buy, hold the two scoring rules: stack-integration fit and scale fit. A vendor that fails either is not a candidate, at any price.
5. Record the decision and its reason in the portfolio memory file.

What "good" looks like: a short build-versus-buy memo per workflow with the five test results, the decision, and the multi-year cost model behind it.

Ownership note: default to owning the parts that carry your judgment. Renting those is how a business ends up unable to operate without its vendors.
