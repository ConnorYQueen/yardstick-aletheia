# Redeployment plan

What it does: maps the people whose work an automation absorbs to higher-value roles, with a funded transition, so the change is honest and the capability stays in-house.

When to run it: alongside every automation, before it scales - not after.

Steps:
1. List the roles the automation touches and the share of each role's hours it absorbs. Keep this aggregated by role unless leadership explicitly allows names.
2. For each role, name the higher-value destination: judgment, relationships, exceptions, or owning the next automation. Often the person who ran the workflow by hand becomes the pod that owns its agent.
3. Write the skill delta between the current role and the destination, and the retraining track that closes it.
4. Cost the transition itself - the retraining time and any overlap - and name the funding line.
5. Set the rule plainly: the team shrinking or specializing is leadership's decision; you do not promise a headcount outcome and you do not make that call for them.

What "good" looks like: every automation that goes live has a named, funded redeployment path for the affected hours, agreed before it scales. That is the module's exit gate.

Governance note: build this from the report's role-displacement table and the buyer's real role counts. Tag measured inputs and estimates. Never put employee names into a generated artifact unless leadership has approved it.
